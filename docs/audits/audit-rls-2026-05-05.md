# CRITICAL

## C-1 — RPC `revive_order_with_stock_check(uuid)` exposée à `anon`/`authenticated`

**État live** : ACL `=X/postgres` (PUBLIC peut EXECUTE) — vérifié via `pg_proc.proacl`.

```
revive_order_with_stock_check
  acl: =X/postgres, postgres=X/postgres, anon=X/postgres,
       authenticated=X/postgres, service_role=X/postgres, supabase_auth_admin=X/postgres
```

La migration `20260427300000_revive_order_with_stock_check.sql` ajoute bien `grant execute on function public.revive_order_with_stock_check(uuid) to service_role;` mais **n'a jamais révoqué le grant PUBLIC par défaut**. En Postgres, `CREATE FUNCTION` accorde `EXECUTE` à PUBLIC par défaut ; `GRANT TO service_role` est additif et ne le retire pas.

**Conséquence** : tout utilisateur authentifié (consumer, producer) — ou même anonyme — peut appeler la RPC via PostgREST. La fonction est `SECURITY DEFINER` et ne contrôle pas l'identité de l'appelant : si un attaquant connaît un `order_id` (UUID — guessing infaisable mais fuite possible via logs / referrer), il peut :

1. Repasser une commande `cancelled+payment_failed` à `pending`.
2. Re-décrémenter le stock des produits associés.
3. Reset `closure_reason` et `cancelled_at`.

L'attaque touche la commande d'un **autre utilisateur** : ce n'est pas une fuite de données mais une corruption transactionnelle (stock, état commande). Combiné à un webhook Stripe rejoué, le comportement devient imprévisible.

**Fix** :
```sql
revoke execute on function public.revive_order_with_stock_check(uuid) from public;
revoke execute on function public.revive_order_with_stock_check(uuid) from anon, authenticated;
-- service_role conserve son grant explicite (cf. 20260427300000)
```

## C-2 — RPC `record_refund_attempt(...)` exposée à `anon`/`authenticated`

Même problème que C-1 — la migration `20260502064800_t102_2b_record_refund_attempt_rpc.sql` ne révoque pas le grant PUBLIC. La fonction est `SECURITY DEFINER` et écrit dans `refund_incidents` + `refund_incident_attempts`, deux tables censées être en mode service-role only.

**Conséquence** : un appelant authentifié peut UPSERT n'importe quelle ligne dans `refund_incidents` avec `(order_id, kind)` arbitraire — empoisonnant le cron retry-failed-refunds (qui dérive son état de cette table). Avec `outcome='succeeded'`, l'attaquant peut silencieusement annuler le suivi d'un refund réel ; avec `classification='permanent'`, il peut court-circuiter un retry légitime à `status='exhausted'`.

**Fix** :
```sql
revoke execute on function public.record_refund_attempt(
  uuid, text, text, uuid, text, text, text, text, text, text, text, text, timestamptz
) from public, anon, authenticated;
grant execute on function public.record_refund_attempt(
  uuid, text, text, uuid, text, text, text, text, text, text, text, text, timestamptz
) to service_role;
```

À auditer aussi : `restore_product_stock_on_order_cancel()` — c'est une fonction trigger (jamais appelée directement) donc l'ACL public est non exploitable (PostgREST n'expose pas les fonctions trigger sans signature). Pas une CRITICAL mais à révoquer pour propreté (LOW-3).

# HIGH

## H-1 — `auth.uid()` et helpers `is_admin()` / `owns_producer()` non-wrappés dans `(select ...)`

29 policies sur 41 utilisent `auth.uid()`, `is_admin()` ou `owns_producer()` directement, sans le wrapper `(select ...)`. Recommandation officielle Supabase : [RLS Performance](https://supabase.com/docs/guides/database/postgres/row-level-security#rls-performance-recommendations).

**Impact** : sur une requête qui scanne N lignes, la fonction est ré-évaluée N fois. Sur les tables volumineuses (`orders`, `order_items`, `audit_logs`, `notifications`), le coût grandit linéairement. Avec wrapping, Postgres caches le résultat en `InitPlan` (1 fois par query).

Tables impactées (toutes : RLS perf dégradée à terme) :
- `users` (3 policies)
- `producers` (4)
- `products` (1 — `owns_producer`)
- `slots`, `slot_rules` (2 chacune — `owns_producer`)
- `orders`, `order_items` (toutes — `auth.uid()` + `owns_producer`)
- `reviews` (4)
- `payouts`, `notifications` (1 chacune)
- `producer_interests`, `producer_invitations` (admin policies)
- `audit_logs`, `disputes`, `refund_incidents`, `refund_incident_attempts` (sub-`select` `from admin_users where id = auth.uid()` — à wrapper)
- `admin_users` (1)
- Storage `product-photos` / `producer-photos` (6 policies — `owns_producer`)

**Fix type** :
```sql
-- Avant
create policy "users self read" on public.users for select using (auth.uid() = id);

-- Après
create policy "users self read" on public.users for select using ((select auth.uid()) = id);
```

Idem pour `(select public.is_admin())` et `(select public.owns_producer(producer_id))` — ce sont des fonctions stables, le résultat se cache. Migration de mise à niveau possible en une seule transaction (drop + recreate de chaque policy).

## H-2 — Sub-queries `EXISTS (...)` inline répétées par-ligne

Cinq policies évaluent un `EXISTS` sur une autre table à chaque row :

- `products public read when producer public` — `EXISTS (SELECT 1 FROM producers p WHERE p.id = products.producer_id AND p.statut = 'public')`
- `slots public read when producer public` — idem
- `slot_rules public read when producer public` — idem
- `order_items via order` — `EXISTS (... FROM orders o WHERE o.id = order_items.order_id AND ...)`
- `reviews consumer insert after completed order` — `EXISTS (... FROM orders o WHERE ...)`

L'EXISTS s'arrête au premier match grâce aux PK indexées, mais reste exécuté par-ligne. Sur `products` (catalogue avec scans), la `producers` est lookupée à chaque row.

**Fix recommandé** : remplacer par une fonction helper `is_producer_public(p_producer_id uuid) RETURNS boolean LANGUAGE sql STABLE SECURITY DEFINER SET search_path = public, pg_temp`. Référencée comme `(select public.is_producer_public(producer_id))`, le résultat est cachable par Postgres et la SD bypass l'éventuelle RLS de `producers`. Pattern aligné avec `owns_producer()`.

Pour `order_items via order` et `reviews ...`, créer un helper similaire `(select public.can_access_order(order_id))` avec la logique parties/owner.

## H-3 — Storage policies sans SELECT — risque d'échec silencieux d'upsert

Migration `20260422100000_storage_policies_for_producers.sql` crée pour chaque bucket (`product-photos`, `producer-photos`) :
- INSERT (owner)
- UPDATE (owner)
- DELETE (owner)

**Mais pas de SELECT**. Skill Supabase : *"Storage upsert requires INSERT + SELECT + UPDATE. Granting only INSERT allows new uploads but file replacement (upsert) silently fails."*

Les buckets sont publics, donc le SELECT public bypasse l'absence de policy pour la lecture lecture seule par URL. Mais l'API `storage.from('product-photos').upload(path, file, { upsert: true })` côté authenticated nécessite SELECT pour le replacement — et cet appel échouera silencieusement (file reste l'ancien) sans erreur retournée.

**Vérification à faire** : grep le code pour `{ upsert: true }` ou `upsert(`. Si l'app fait du upsert sur ces buckets, ajouter :
```sql
create policy "product-photos owner select"
  on storage.objects for select to authenticated
  using (
    bucket_id = 'product-photos'
    and (select public.owns_producer(((storage.foldername(name))[1])::uuid))
  );
-- idem producer-photos
```

# MEDIUM

## M-1 — Pas de `FORCE ROW LEVEL SECURITY` sur les tables sensibles

`alter table ... force row level security` est `false` partout. Conséquence : le `table owner` (postgres) bypasse RLS. Pour Supabase, `service_role` a aussi `BYPASSRLS` et continuera à fonctionner. La défense en profondeur consiste à `force` les tables vraiment sensibles, pour qu'une session SQL-Editor (qui se connecte en `postgres`) doive explicitement SET ROLE pour bypass.

Tables candidates (sensibles, jamais accédées directement en `postgres`) :
- `audit_logs` (forensique RGPD/PCI)
- `disputes`
- `refund_incidents` / `refund_incident_attempts`
- `payouts`
- `email_change_otp_codes`, `email_change_undo_tokens` (secrets)
- `webhook_events_processed`

Risque actuel : faible (accès SQL Editor limité aux admins humains via Dashboard). Defense-in-depth.

## M-2 — Policies admin manquantes sur tables write-only-service_role

Aucune policy admin sur :
- `users` (lookup users impossible depuis admin authenticated)
- `orders`, `order_items` (consultation ops impossible sans service_role)
- `payouts` (suivi virements)
- `products`, `slots` (modération catalogue)
- `notifications` (debug)

Tout passe actuellement par service_role côté back-office. **Acceptable** par convention projet, mais à documenter dans `METHODOLOGY.md` : *"toute consultation admin de tables sans `*_admin_*` policy passe par `createSupabaseAdminClient`"*. Sinon risque de dette future si quelqu'un branche une page admin via le client authenticated et obtient des résultats vides sans erreur.

## M-3 — Drift T-241 : migration locale non appliquée en prod

L'audit live confirme que `public.producers` **n'a pas** les colonnes `declaration_indicateurs_veracite_at`, `declaration_indicateurs_snapshot`, `declaration_indicateurs_wording_version`, et que la fonction `update_producer_onboarding(uuid, ...)` **n'existe pas**. Le commit `e352a63` est mergé sur master (cf. recent commits) mais la migration `20260504100000_t241_declaration_veracite_persistance.sql` n'est ni dans `supabase_migrations` ni reflétée par le schéma live. Le code applicatif déployé qui appelle cette RPC plantera.

**Sous-finding** : quand T-241 sera apply, la nouvelle RPC `update_producer_onboarding` est :
- `SECURITY DEFINER`
- Pas de garde `auth.uid() = p_user_id` (le commentaire le dit explicitement, en s'appuyant sur "appelée par service_role uniquement")
- Aucun `revoke execute from public` dans la migration

**Conséquence anticipée** : par défaut PUBLIC peut EXECUTE, donc tout authenticated peut appeler la RPC avec un `p_user_id` arbitraire et **modifier la fiche producteur d'un autre user**. Ce serait une CRITICAL si appliqué tel quel.

**Fix avant apply** :
```sql
-- Ajouter en tête de fonction (version recommandée)
if auth.uid() is null or auth.uid() is distinct from p_user_id then
  raise exception 'Not authorized'
    using errcode = '42501';
end if;

-- Et après le create
revoke execute on function public.update_producer_onboarding(...) from public;
grant execute on function public.update_producer_onboarding(...) to service_role;
```

## M-4 — `producer_interests public insert` ouvert sans rate limit DB

```sql
create policy "producer_interests public insert"
  on public.producer_interests for insert
  to anon, authenticated
  with check (true);
```

`with_check = true` accepte tout. Le formulaire `/devenir-producteur` est rate-limité côté app (`lib/producer-interests/upsert-interest.ts` à vérifier). Au niveau DB, un attaquant peut spammer la table jusqu'à saturation. Le `UNIQUE (email)` (mig 20260428300000) limite à 1 row par email mais pas l'INSERT churn.

**Mitigation possible** : ajouter une check sur la longueur des champs (déjà couvert côté app) ou un rate-limit Postgres via `pg_cron` (purge des rows `created_at > now()-1h` au-delà d'un seuil par IP — mais on ne stocke pas l'IP). Acceptable en l'état si le rate limit applicatif est solide.

## M-5 — Policy `disputes_service_role_all` redondante

```
schemaname=storage, tablename=disputes
policyname=disputes_service_role_all  cmd=ALL  roles={service_role}  qual=true  with_check=true
```

Cette policy n'est dans aucune migration du repo — elle a été ajoutée via Dashboard. Le `service_role` bypasse RLS nativement (`BYPASSRLS`), donc cette policy n'a aucun effet. Elle peut induire en erreur (donne l'impression qu'on a besoin d'une policy pour service_role).

**Fix** : `drop policy "disputes_service_role_all" on public.disputes;` puis snapshotter le résultat dans une migration `repo-aligned`.

## M-6 — Index ACL `=X/postgres` sur toutes les fonctions = grant PUBLIC implicite

Conséquence générale de C-1/C-2 : 14 fonctions ont l'ACL PUBLIC EXECUTE par défaut. Pour les fonctions avec garde interne (`create_order_with_items`, `delete_user_account`) c'est acceptable — elles checkent `auth.uid()`. Pour les fonctions trigger (`compute_order_commission`, `set_order_code`, `set_updated_at`, `slot_rules_set_updated_at`, `enforce_user_exclusive`, `restore_product_stock_on_order_cancel`, `generate_order_code`), aucune n'est exposable via PostgREST (pas appelable comme RPC), donc sans risque pratique. Mais propreté défensive.

**Recommandation** : ajouter une migration de cleanup qui révoque PUBLIC sur toutes les fonctions sauf celles intentionnellement publiques (`is_admin`, `owns_producer`, `search_producers`, `create_order_with_items`, `delete_user_account`).

# LOW

## L-1 — Policies `to public` au lieu de `to authenticated`

Quand le check est `auth.uid() = ...`, scoper à `to authenticated` évite l'évaluation pour anon (qui aurait `auth.uid() = NULL` → null=val → false → refus). Cosmétique, perf marginal sur fortes charges anon.

Concernés : 22 policies (toutes celles avec `roles={public}` qui font un check d'identité). Liste exhaustive disponible via `pg_policies`.

## L-2 — Policy admin pour `users` : prévue ?

Question ouverte : un admin doit-il pouvoir lire `public.users` via PostgREST en authenticated ? Si oui, ajouter `users admin read` sur le pattern `is_admin()`. Si non (service_role only), documenter explicitement dans M-2.

## L-3 — `restore_product_stock_on_order_cancel` exposable PUBLIC

Fonction trigger, pas appelable via PostgREST (pas de signature standard). ACL PUBLIC inoffensive en pratique. Révoquer pour propreté (cf. M-6).

## L-4 — Index `(statut, source)` manquant sur `producer_interests`

L'admin liste les leads filtrés sur `statut='new'` et/ou `source='formulaire_public'`. Index sur `statut` seul existe (`producer_interests_statut_idx`). Volume actuel négligeable, à garder en tête si les leads montent.

# Appendice — Isolation www / pro / admin

Le modèle actuel :

- **www (consumer)** : auth Supabase classique → `public.users` (`roles` array contient `'consumer'`).
- **pro (producer)** : même auth Supabase → même ligne `public.users` avec `roles` contenant `'consumer'` et `'producer'`. Pas de séparation auth — l'isolation www/pro est purement applicative (subdomain + middleware).
- **admin** : auth Supabase distincte → `public.admin_users` (jamais dans `public.users`, garanti par trigger `enforce_user_exclusive`).

**RLS reflète ces privilèges correctement** :
- `is_admin()` interroge `admin_users.id = auth.uid()` (security definer, search_path verrouillé).
- `owns_producer(p_producer_id)` interroge `producers.user_id = auth.uid()` (idem).
- Pas de policy "producer-only" qui scoperait sur `'producer' = ANY(users.roles)` — l'access producteur est implicite via owns_producer.

**Trous potentiels** :
- Aucun. La cumulativité consumer + producer n'introduit pas de leak parce que la lecture des données producteur passe toujours par `owns_producer(producer_id)` qui est tied au `user_id` du producteur, pas au rôle générique.
- L'isolation **subdomain** (www vs pro vs admin) n'est pas et ne peut pas être enforcée par RLS — Postgres ne voit pas le `Host:` HTTP. Bien fait au niveau middleware Next.js.

**Conclusion isolation** : le modèle 3 sub-domains est correctement reflété. Pas de finding à ce niveau.
