## Vérification server action T-241 : OK

**Statut** : aucune modification TS nécessaire — la garde `auth.uid() === p_user_id` est implicitement enforced par la chaîne d'appel.

**Constat** :
- Unique call site dans `app/` : `app/(producer)/invitation/_actions/complete-onboarding.ts:145-164`.
- L'argument `p_user_id` est passé exclusivement comme `session.id` (ligne 146).
- `session` provient de `getSessionUser()` (`lib/auth/session.ts:16-41`) qui invoque `supabase.auth.getUser()` — appel qui revalide le JWT côté serveur Supabase Auth (pas un cookie potentiellement falsifiable, contrairement à `getSession()`).
- Aucun champ `user_id` n'est lu depuis `formData` (cf. lignes 26-46 — la liste exhaustive des champs lus exclut tout `user_id`).
- La server action retourne `{ error: "Session expirée" }` si `getSessionUser()` retourne null (ligne 24).

**Conséquence** : il n'existe aucun chemin où `p_user_id` pourrait être ≠ de l'identité authentifiée. Combiné avec l'ACL `service_role only` posée par le patch T-241 (lot 6), la RPC est doublement protégée :
- Niveau ACL : seul `service_role` peut EXECUTE → un attaquant ne peut pas appeler la RPC directement via PostgREST.
- Niveau applicatif : la server action ne propage jamais une valeur de `p_user_id` autre que celle issue du JWT validé.

Aucun diff TS appliqué.

## Arbitrages tranchés

### MEDIUM-2 — Policies admin manquantes (orders / payouts / products / slots / notifications / users)

**Décision** : convention « admin = service_role » appliquée de fait pour ces tables. **Aucune migration créée**, **aucun fichier de méthodologie créé** dans le repo.

**Justification** : la convention est portée par le skill supabase (re-chargé à chaque session) et par les migrations historiques qui n'ont jamais ajouté de policy admin sur ces tables — c'est un choix architectural cohérent depuis l'origine. Documenter dans un fichier interne au repo (`METHODOLOGY.md`) ferait double-emploi avec le skill et risquerait de diverger.

**Implication** : tout futur dev consommant ces tables côté admin doit utiliser `createSupabaseAdminClient()` (service_role). Les helpers `lib/supabase-admin.ts` portent déjà le pattern. Si une page admin authenticated est ajoutée par erreur, elle retournera 0 rows silencieusement — symptôme reconnaissable au QA.

### MEDIUM-4 — Rate-limit DB sur producer_interests INSERT

**Décision** : aucun rate-limit DB ajouté. Le `UNIQUE(email)` existant (migration 20260428300000) est suffisant pour le volume actuel. **Aucune migration créée**.

**Justification** : volume actuel ~10 leads, croissance prévisible faible (acquisition producteurs = canal de niche, pas de viralité). Un trigger BEFORE INSERT ou un cron `pg_cron` serait du surcuit pour un risque théorique. Le rate-limit applicatif (Next middleware / route POST `/api/producer-interests`) reste la première barrière contre le spam — déjà en place et suffisant.

**Re-évaluation** : si le volume passe au-delà de ~100 leads/mois ou si un incident spam est constaté, ré-ouvrir un chantier dédié avec arbitrage seuils + mécanisme.

### LOW-2 — Policy admin sur `users`

**Décision** : pas de policy admin authenticated. La table reste service_role only pour les chemins admin.

**Justification** : aligné MEDIUM-2 — le back-office admin consomme `public.users` exclusivement via `createSupabaseAdminClient()`. Pas besoin d'exposer la table à la session admin authenticated via PostgREST. La séparation `users` ↔ `admin_users` (cf. migration 20260421100000) reste totale.

### LOW-5 — Drift `supabase_migrations` vs repo

**Décision** : backfill manuel léger des 35 entrées manquantes via script one-shot (`scripts/maintenance/backfill-supabase-migrations-2026-05-05.sql`) + adoption progressive de `supabase migration up --linked` pour toute nouvelle migration.

### Lot 6 — Patch T-241 update_producer_onboarding

**Finding** : MEDIUM-3.
**Fichier modifié (pas de nouveau fichier)** : `supabase/migrations/20260504100000_t241_declaration_veracite_persistance.sql`

Couvert :
- En-tête de migration enrichi : note d'avertissement audit, contrat ACL service_role only, procédure d'ouverture future à authenticated.
- Comment `update_producer_onboarding` mis à jour pour refléter le verrouillage ACL.
- Ajout en fin de migration : `REVOKE EXECUTE ... FROM public, anon, authenticated;` + `GRANT EXECUTE ... TO service_role;`.

**PAS appliqué** : la garde interne `auth.uid() = p_user_id` n'a pas été ajoutée. Justification : la server action `complete-onboarding.ts` appelle la RPC via le client admin (service_role), qui n'a pas de JWT context (auth.uid() = NULL) — un guard auth.uid() casserait la chaîne d'appel et exigerait une modification applicative TS, hors scope. La défense réelle vient de la combinaison ACL service_role only + dérivation server-side de `p_user_id` depuis `getSessionUser()` (cf. section « Vérification server action T-241 » ci-dessus).

Points d'attention apply :
- Le fichier T-241 n'a pas encore été appliqué en prod (audit confirme : pas de `declaration_indicateurs_*`, pas de fonction `update_producer_onboarding`). Cette migration patchée doit être appliquée avant tout déploiement de code applicatif consommant la RPC.
- Vérifier post-apply : `SELECT has_function_privilege('authenticated', 'public.update_producer_onboarding(...)', 'EXECUTE')` doit retourner `false` ; idem `'anon'`.
- Si T-241 a été apply entre cet audit et ce fix (peu probable, à vérifier via `supabase_migrations`), exécuter le patch ACL séparément :
  ```sql
  revoke execute on function public.update_producer_onboarding(uuid, text, text, text, text, text, text, text, text, text, text, text, text, boolean, text)
    from public, anon, authenticated;
  grant execute on function public.update_producer_onboarding(uuid, text, text, text, text, text, text, text, text, text, text, text, text, boolean, text)
    to service_role;
  ```
