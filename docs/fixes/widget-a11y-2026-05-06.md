# Audit a11y DistanceWidget — pattern Disclosure (T-273)

> Session 2026-05-06. Scope : `app/(public)/producteurs/[slug]/_components/DistanceWidget.tsx`.
>
> Cet audit clôt **T-273**. Toute dette résiduelle reste portée par **T-256** (refacto disclosure transverse, dette a11y consciente).

---

## Constat avant T-273

Le composant disclosure du widget distance était fonctionnellement correct (toggle ouvert/fermé, focus visible, tap target ≥ 44 px) mais ne suivait pas le pattern WAI-ARIA *Disclosure* :

- `aria-expanded` absent sur les boutons toggle (`CollapsedButton`, `CollapseLink`).
- `aria-controls` absent — pas de lien explicite bouton ↔ panneau.
- Pas d'`id` sur le conteneur déployé.
- Lien `Masquer` sans `aria-label` descriptif (le textContent seul est ambigu hors contexte).

Conséquence pratique : un screen reader (NVDA, VoiceOver) ne peut pas annoncer l'état "collapsed/expanded" ni faire le saut bouton → panneau via la commande de navigation Skipper.

---

## Modifications T-273 (cas B — fix mineur dans le même commit)

### 1. Constante de panneau

```ts
const PANEL_ID = "distance-widget-panel";
```

### 2. Bouton replié — `CollapsedButton`

```tsx
<button
  type="button"
  onClick={onClick}
  disabled={disabled}
  aria-expanded={false}
  aria-controls={PANEL_ID}
  ...
>
```

### 3. Bouton "Masquer" — `CollapseLink`

```tsx
<button
  type="button"
  onClick={onClick}
  aria-expanded={true}
  aria-controls={PANEL_ID}
  aria-label="Masquer le détail de la distance"
  ...
>
  Masquer
</button>
```

`aria-label` explicite car le textContent `Masquer` seul est ambigu (un screen reader sortant la page ne sait pas de quoi).

### 4. Panneau déployé — 3 sites

Le widget rend 3 variantes de panneau selon l'état :
- invite déployée sans session (formulaire géoloc + CP + RGPD),
- `DistanceResult` (distance + barre comparative),
- `DistanceOutOfReach` (hors zone circuit court).

Les 3 portent désormais `id={PANEL_ID}`. Multi-trigger (mobile + desktop dans `DistanceResult`) sur un même panneau est ARIA-valide.

## Choix conscient — pas de focus management explicite

Le pattern Disclosure WAI-ARIA recommande de laisser le focus là où il est au toggle (l'utilisateur clavier reste sur le bouton qu'il vient d'activer ; le contenu est lu post-render via `aria-live` ou simple navigation séquentielle).

On ne fait pas de `setFocus()` automatique post-expansion :
- pas de surprise pour l'utilisateur (son focus ne saute pas),
- compatible avec les comportements natifs lecteur d'écran (NVDA annonce "expanded" puis lit le contenu en navigation rapide F),
- évite les bugs subtils (focus vers un élément qui n'est pas encore monté, focus capturé par un overlay).

Refactor disclosure plus profond (focus trap, retour focus au collapse, ESC pour replier) → **T-256** (déjà tracée comme dette consciente).
